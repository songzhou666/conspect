# Conspect Skill 反模式说明文档

> 本文档列举了使用 conspect Skill 过程中常见的错误用法及其改进方案，帮助用户避免踩坑、提升报表生成效率与质量。

---

## 1. 绕过状态机直接渲染

**反模式描述：** 用户在数据预处理或设计阶段尚未完成时，直接要求 conspect Skill 生成最终图表/报表，跳过分析、设计、审核等环节。

**典型案例：**
- "不用分析了，直接给我出一张销售额趋势图"
- "数据我已经看过了，跳过设计阶段，开始渲染"

**问题分析：**
- conspect Skill 的状态机（分析 → 设计 → 实现 → 验证 → 完成）是质量保证的核心机制
- 跳过分析阶段会导致图表选型失当、数据维度遗漏
- 跳过设计阶段会造成布局混乱、可读性差
- 跳过审核阶段可能输出存在缺陷的报表

**改进方案：**
- 坚持走完整流程，每个阶段都有明确的输出产物
- 如果对某个阶段有预判，可以在该阶段的交互中提供输入（如"分析时重点关注XX指标"），而非跳过
- 使用 `--fast-track` 参数仅在数据质量极高、需求极其明确时可适度加速，但仍不可跳过质量审核

---

## 2. 前端承载原始数据

**反模式描述：** 用户要求将原始明细数据直接嵌入前端 HTML/JavaScript 代码中，让浏览器端负责数据处理和聚合。

**典型案例：**
- "把 10 万行销售明细写到 JS 变量里，前端做个汇总表"
- "让前端 ECharts 自己计算各维度汇总"

**问题分析：**
- 违反"原始明细不上前端"的核心设计原则
- 前端承载大量原始数据会导致页面加载缓慢、内存占用飙升
- 数据处理逻辑暴露在前端存在数据泄露风险
- 浏览器端的计算能力有限，大数据量聚合可能导致页面卡死

**改进方案：**
- 后端（Python/Node.js 预处理脚本）完成数据清洗、聚合、计算
- 前端只接收渲染所需的汇总数据和图表配置
- 遵循后端全量处理、前端轻量渲染的分层架构
- 使用 `conspect data summarize` 命令自动生成聚合数据

## 3. 人工指定图表类型

**反模式描述：** 用户强行指定某张图表必须使用某种特定类型，而不接受 AI 的推荐选型。

**典型案例：**
- "不要柱状图，就要用雷达图来展示月度趋势"
- "所有对比必须用饼图，不要问为什么"

**问题分析：**
- 雷达图不适合展示时间序列趋势，信息密度低、可读性差
- 饼图在类别过多（>5 类）时难以直观比较
- 人工指定可能忽略数据特征与图表类型的匹配关系
- 导致生成的报表难以理解，反而需要反复返工

**改进方案：**
- 充分信任 AI 基于数据特征（维度基数、数值分布、时间序列等）的选型建议
- 如果对某类数据有特殊的呈现偏好，可以在 DESIGN 阶段以"推荐 + 微调"的方式表达
- 例如："对于占比数据优先考虑饼图，但如果超过 6 个类别请改用条形图"
- 在质量审核环节可以对图表类型提出修改意见，而非在最初强行指定

---

## 4. 混用桌面 BI 工具

**反模式描述：** 用户要求将 conspect Skill 的产出导入到 PowerBI / Tableau 等桌面 BI 工具中继续处理，或要求输出专有格式。

**典型案例：**
- "生成的报表能不能导出成 PowerBI 的 .pbix 格式"
- "我想在 Tableau 里继续加工，输出 .twb 文件"

**问题分析：**
- conspect Skill 遵循"零客户端"约束，产出物应为纯 Web 端可直接访问的格式
- 桌面 BI 工具需要用户额外安装软件，违背了即开即用的理念
- 专有格式（.pbix / .twb）限制了产出的可移植性和分享便捷性
- 增加学习和使用成本，不适合运营人员/客户等非技术用户

**改进方案：**
- conspect Skill 默认输出独立 HTML 文件（含内嵌 CSS/JS），任何浏览器即可打开
- 支持导出为 PDF、PNG 等通用格式用于分享
- 如需进一步分析，可以从 conspect 输出的聚合数据（JSON/CSV）中导出到其他工具
- 如果确实需要对接 BI 工具，建议在 VERIFY 阶段的输出中额外提供数据源文件（CSV/Excel）

## 5. 单数据源局限

**反模式描述：** 用户只上传一个数据文件就期望生成全方位、多维度的丰富报表。

**典型案例：**
- "我只有一个门店的销售数据，帮我做一份全公司经营分析报告"
- "只有本月数据，但我要同比环比分析"

**问题分析：**
- 单一数据源能支撑的分析维度有限，无法覆盖多维度、多指标的完整分析
- 缺少对照数据（如历史数据、目标数据、行业基准），分析结论缺乏参照
- 容易产出"有图无结论"的报表，信息量不足

**改进方案：**
- 建议至少提供 2-3 个关联数据源（如：销售数据 + 目标数据 + 历史同期数据）
- 利用 conspect 的数据融合能力进行多源交叉分析
- 如果确实只有一个数据源，应在 ANALYZE 阶段说明约束，AI 会自动调整分析范围和深度
- 考虑使用外部参考数据（行业报告、公开数据集）辅助分析

---

## 6. 跳过质量审核

**反模式描述：** 用户在报表生成后认为质量审核环节多余，要求直接交付。

**典型案例：**
- "我看着没问题，审核跳过吧"
- "审核太慢了，先把报表发出去"

**问题分析：**
- 质量审核是 conspect Skill 的阻断性环节——不通过审核不得交付
- 人工目视检查可能遗漏数值精度、色盲友好、跨屏适配等细节问题
- 审核环节包含自动化检测（数据一致性校验）和人工复核（视觉体验评估）
- 跳过审核可能导致交付的报表存在数据错误或体验缺陷

**改进方案：**
- 理解质量审核的必要性，将其视为报表质量的"最后一道防线"
- 在审核阶段可以做最后的微调修改，避免交付后返工
- 如果审核标准过于严格，可以在项目初期调整审核规则配置
- 审核不通过时，根据审核报告逐条修改，而非直接忽略

## 7. 与其他 Skill 产物命名冲突

**反模式描述：** 用户在同一项目中同时使用 conspect 和 reqplan-v3（或其他 Skill）时，输出文件使用相同的命名前缀，导致文件路径混淆。

**典型案例：**
- conspect 输出 `_analysis.md`，reqplan-v3 也输出 `_analysis.md`，相互覆盖
- 两个 Skill 的设计文档同名，分不清哪个是哪个

**问题分析：**
- 多个 Skill 在同一项目中运行时，默认产物名称可能冲突
- 同名文件被覆盖后，审核或追溯时找不到正确版本
- 增加认知负担，用户需要手动区分文件归属

**改进方案：**
- conspect Skill 的所有产物统一使用 `_cs-` 前缀（如 `_cs-analysis.md`、`_cs-design.md`）
- reqplan-v3 的产物使用 `_rp-` 前缀或其他有区分的命名规则
- 在项目初始化时配置 output_prefix 参数，确保唯一性
- 产物清单中明确标注每个文件对应的 Skill 名称

---

## 8. 只分析第一个 Sheet

**反模式描述：** 用户上传的 Excel 文件包含多个 Sheet，但 Analyzer Agent 只读取了第一个 Sheet 就开始分析，遗漏了其他关联数据。

**典型案例：**
- "上传了一个 Excel，里面有销售明细、产品信息、客户信息等多个 Sheet，但报告只用到了第一个 Sheet"
- "分析报告只覆盖了部分数据，其他 Sheet 中的数据完全没有提及"

**问题分析：**
- 数据流水线要求"对每个 Excel 文件，遍历所有 Sheet"，但 Analyzer 的前置检查中未强制枚举 Sheet 列表
- Analyzer 默认使用 `pd.read_excel()` 只读取第一个 Sheet，未主动检查是否还有其他 Sheet
- 缺少 Sheet 完整性校验点，导致遗漏未被及时发现
- 多 Sheet 间的关联分析（VLOOKUP 类操作）被完全跳过，分析结论片面

**改进方案：**
- Analyzer Agent 的「前置检查」必须强制执行 Sheet 枚举：使用 `pd.ExcelFile` 列出所有 Sheet 名称并记录到分析报告
- 分析报告中「数据概览」部分必须包含 "Sheet 列表" 一节，明确列出所有 Sheet 及其行/列数
- 对于关联表（如产品信息表、客户信息表），需评估是否需要横向合并以丰富维度
- 质量审核 Agent 将"数据源完整性"列为独立的审核维度，未覆盖全部 Sheet 直接打回
- 如果某些 Sheet 确实不需要分析，必须在报告中注明排除原因

## 9. 忽略数据安全

**反模式描述：** 用户要求在前端代码中包含原始明细数据，或在报表中展示敏感字段（如手机号、身份证、银行账号）。

**典型案例：**
- "在工具提示里显示每个客户的手机号"
- "把详细的客户地址数据写到 HTML 里"

**问题分析：**
- 前端代码对任何访问者可见，原始明细数据直接暴露
- 数据脱敏不到位可能违反数据保护法规（如个保法、GDPR）
- 即使报表仅供内部使用，数据泄露的风险依然存在
- 浏览器 DevTools 可轻易查看所有前端数据

**改进方案：**
- 严格遵守"原始明细不上前端"的数据安全红线
- 在前端只展示脱敏后的聚合数据（汇总、占比、趋势）
- 如需查看明细，设计独立的"详情下钻"机制，通过后端 API 按需获取
- 对于必须展示的敏感字段，使用脱敏处理（如手机号 138****1234）
- 定期检查生成的 HTML 文件中是否意外包含了原始数据
