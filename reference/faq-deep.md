## [数据处理]

### Q1: conspect Skill 支持的最大文件大小和行数是多少？

**A:** conspect Skill 对输入数据有以下约束：

| 数据源类型 | 建议上限 | 硬限制 | 说明 |
|-----------|---------|-------|------|
| CSV 文件 | 50 MB / 50 万行 | 200 MB / 200 万行 | 超过建议上限后预处理时间显著增加 |
| Excel (.xlsx) | 20 MB / 30 万行 | 100 MB / 100 万行 | Excel 格式本身存在解析开销 |
| JSON 文件 | 30 MB / 10 万节点 | 100 MB / 50 万节点 | 嵌套结构会放大解析复杂度 |
| 数据库直连 | 无硬限制 | 受查询超时限制（默认 300s） | 建议在 SQL 层面做预聚合 |

**超限处理策略：**

- 文件超过建议上限时，系统自动触发采样机制（默认保留 10% 行数 + 尾部分布片段）
- 超过硬限制时，系统拒绝加载并提示用户拆分文件
- 对于 Excel 文件，超过 100 万行建议拆分为多个 Sheet 或改用 CSV 格式

**最佳实践：**
- 大数据量场景优先使用 CSV 格式（解析效率最高）
- 在数据源端完成聚合后再上传（如：月汇总代替日明细）
- 使用数据预览功能在大文件加载前快速查看结构

### Q2: Excel 编码或格式异常怎么处理？

**A:** conspect Skill 内置了多层容错机制处理异常 Excel 文件：

**自动检测流程**
1. **编码检测** — 自动识别 UTF-8、GBK、GB2312、ISO-8859-1、Shift-JIS 等编码
2. **格式校验** — 检查文件头魔数（magic number），确认是否为有效 .xlsx/.xls
3. **结构分析** — 解析 Sheet 结构、列类型推断、空行/空列检测

**已知问题及处理**
| 异常类型 | 系统行为 | 用户应对 |
|---------|---------|---------|
| 编码不兼容 | 自动尝试常见编码回退，输出警告 | 另存为 UTF-8 格式重新上传 |
| 损坏的 .xlsx 文件 | 尝试使用 openpyxl 恢复模式读取 | 使用 Excel 打开并另存为新文件 |
| 日期格式混乱 | 自动检测日期模式并归一化 | 确保日期列格式统一 |
| 科学计数法显示 | 自动将数值列转为字符串精度 | 提前设置列格式为"文本" |
| 混合类型列 | 该列转为字符串类型，丢失数值计算能力 | 确保同列数据类型一致 |

> 如果系统给出"格式异常"警告，建议先用 Excel 打开文件 -> "另存为" -> 选择 .xlsx 格式 -> 重新上传。

### Q3: 如何处理含合并单元格的 Excel？

**A:** 合并单元格是 Excel 数据分析中最常见的复杂场景之一，conspect Skill 提供了专门的预处理策略。

**自动检测机制**
系统在数据加载阶段自动扫描：
1. 检测合并单元格的位置和范围
2. 判断合并类型（标题合并 / 分组合并 / 格式合并）
3. 根据合并类型选择填充策略

**合并单元格处理策略**

| 合并类型 | 系统默认行为 | 高级参数控制 |
|---------|-------------|-------------|
| **标题合并**（表头区域） | 保留合并结构，解析为层级表头 | `--header-merge keep` |
| **分组合并**（如"部门"列有多行合并） | 自动向下填充（fill down） | `--fill-method down / copy / interpolate` |
| **格式合并**（仅视觉合并，数值无意义） | 拆分为独立行，每个单元格独立 | `--format-merge split` |
| **跨行合并**（跨多行多列） | 拆分为行列笛卡尔积 | `--cross-merge expand` |

**特殊情况处理**

情况 1：合并单元格内有换行符
- 系统自动按换行拆分，或根据参数 `--newline-split concat/split` 选择合并/拆分

情况 2：嵌套合并
```
| 部门        | 2024 Q1       | 2024 Q2       |
| (合并4行)   | 销售额 利润   | 销售额 利润   |
```
- 系统自动将双层表头展平为：`部门 | 2024Q1_销售额 | 2024Q1_利润 | 2024Q2_销售额 | 2024Q2_利润`

情况 3：合并单元格内公式引用
- 系统保留公式计算结果，忽略公式表达式本身

**最佳实践**
- **上传前预处理** — 如果合并单元格结构过于复杂，建议在 Excel 中取消合并并用"填充-向下"功能预处理
- **使用 `--merge-report` 参数** — 系统输出合并单元格处理报告，方便人工核查
- **对合并结果有疑虑时** — 在分析阶段检查解析后的数据框预览

### Q4: 多 Sheet 的关联分析怎么做？

**A:** conspect Skill 原生支持跨 Sheet 关联分析，无需手动合并数据。

**使用步骤**

步骤 1：加载多 Sheet 文件
在 ANALYZE 阶段，系统会自动加载用户上传的 Excel 文件中的所有 Sheet。也可以告诉 AI 指定需要分析的 Sheet。

步骤 2：声明关联关系
在 ANALYZE 阶段系统自动推断键列（key column），也可以通过配置明确声明：

```yaml
relationships:
  - source: "订单明细"
    target: "客户信息"
    type: left_join
    keys:
      - ["客户ID", "客户编号"]
  - source: "订单明细"
    target: "产品目录"
    type: left_join
    keys:
      - ["产品ID", "产品编码"]
```

步骤 3：执行关联分析
关联关系声明后，系统将自动执行：
1. 自动推断关联键（基于列名相似度和数据类型匹配）
2. 选择最优连接类型（left/inner/full）
3. 处理重复键和多对多关系
4. 输出关联后的统一数据视图

**关联分析支持的模式**

| 模式 | 描述 | 适用场景 |
|------|------|---------|
| 主从关联 | 一个主 Sheet 关联多个维度 Sheet | 订单 + 客户 + 产品 |
| 星型关联 | 多个事实 Sheet 共享维度 Sheet | 销售 + 退货 + 预算，共享时间/区域维度 |
| 血缘关联 | Sheet 间存在间接关联（A-B-C） | 订单明细 -> 发货单 -> 物流信息 |
| 时间关联 | 以时间为键关联不同时间粒度数据 | 日销售数据 + 月目标数据 |

**注意事项**
- Sheet 数量建议不超过 10 个（超过后关联图复杂，分析效率下降）
- 关联键的列名尽量一致（如都用"客户ID"而非"客户编号"），减少手动映射
- 多 Sheet 关联后的数据量可能是单个 Sheet 的数十倍，注意 Q1 中的文件大小限制

### Q5: 如何处理全角/半角字符混排的数据？

**A:** 数据中的全角/半角混排可能导致维度分组错误（如"张三"和"张 三"被识别为两个不同的值）。

**系统自动处理：**
- 在数据清洗阶段自动将全角字母/数字转为半角（如全角数字"１２３" -> "123"）
- 对中文字符的全角/半角空格进行归一化（移除首尾空格，中间空格保留）
- 对列名中的全角字符自动转半角（避免 SQL/JSON 兼容性问题）

**用户自查：**
- 可在 ANALYZE 阶段要求 AI 检查数据中的全角/半角混排情况
- 系统会在分析报告中标注发现的混排字段和归一化处理数量

## [质量管理]

### Q6: 质量审核不通过如何处理？

**A:** 质量审核是 conspect Skill 的阻断性环节，不通过则不允许进入交付阶段。

**审核未通过的常见原因**

| 类别 | 典型问题 | 出现频率 |
|------|---------|---------|
| 数据准确性 | 图表数值与源数据不一致、汇总计算错误 | 最高 |
| 图表选型 | 图表类型与数据特征不匹配（如用饼图展示趋势） | 较高 |
| 可读性 | 标签重叠、字号过小、配色对比度不足 | 中等 |
| 完整性 | 缺少图例、单位标注、数据来源说明 | 中等 |
| 色盲友好 | 使用红绿配色区分关键数据 | 较低 |
| 跨屏适配 | 在移动端布局错乱 | 较低 |

**处理流程**

1. **读取审核报告** — 系统会输出详细的 `_cs-qa-{phase}.md`（按阶段生成的审核报告），逐条列出问题
2. **定位问题阶段** — 根据问题类型确定需要回退的环节：
   - 数据问题 -> 回退到 ANALYZE 阶段重新处理数据
   - 设计问题 -> 回退到 DESIGN 阶段修改布局/配色/图表类型
   - 实现问题 -> 回退到 IMPLEMENT 阶段修改渲染代码
3. **逐条修复** — 按审核报告列出的问题逐条修改
4. **重新提交审核** — 修复完后再次触发质量审核
5. **最多重试机制** — 同一轮审核最多重试 3 次，超过后系统输出"重大缺陷分析"建议重新启动流程

**强制通过**
在极少数情况下（如内部临时查看），管理员可以使用 `--force-deliver` 参数强制跳过审核，但系统会在报表上添加水印 `[未经质量审核]`。

## [系统集成]

### Q7: 能不能和其他 Skill 同时使用？

**A:** 可以，但需要注意协作模式和产物隔离。

**支持的协作模式**

| 协作 Skill | 协作阶段 | 协作方式 |
|-----------|---------|---------|
| reqplan-v3 | ANALYZE -> DESIGN | reqplan-v3 输出需求分析和设计文档 -> conspect 据此生成报表 |
| scout | ANALYZE | scout 采集的页面结构数据可作为 conspect 的数据源之一 |
| fepilot | VERIFY | fepilot 对 conspect 生成的前端页面做缺陷检测 |
| parse | ANALYZE | parse 产出的 Bug 数据可作为 conspect 统计报表的输入 |

**需要注意的问题**

1. **产物命名冲突** — 不同 Skill 可能输出同名文件（如 `_analysis.md`），参见反模式第 7 条
2. **接力棒冲突** — 两个 Skill 同时运行时可能互相覆盖接力棒文件，参见 Q8
3. **上下文膨胀** — 同时加载多个 Skill 的规则文件会占用大量 token，建议分阶段使用
4. **依赖顺序** — 如果 conspect 依赖其他 Skill 的产出，务必先执行依赖 Skill

**推荐实践**
```
阶段一：reqplan-v3 分析需求 -> 输出 _rp-analysis.md
阶段二：conspect 读取 _rp-analysis.md -> 生成报表
阶段三：fepilot 扫描 conspect 的 HTML 产出 -> 输出缺陷报告
```

### Q8: 接力棒文件冲突了怎么办？

**A:** 接力棒（baton）文件是 Skill 状态机的核心状态文件，冲突可能导致流程混乱。

**冲突场景**

| 场景 | 表现 | 风险等级 |
|------|------|---------|
| 两个 Skill 同时写同一个接力棒 | 内容被交替覆盖，状态混乱 | 高 |
| 手动编辑接力棒 | 格式错误、状态不连续 | 中 |
| 接力棒文件被误删除 | Skill 从 INIT 阶段重启 | 中 |

**解决方案**

方案一：使用独立的接力棒路径（推荐）
每个 Skill 使用独立的接力棒文件路径：
- conspect -> `.agent/harness/_cs-baton.md`
- reqplan-v3 -> `.agent/harness/_rp_baton.json`
- scout -> `.agent/harness/_scout_baton.json`

方案二：串行执行（最安全）
确保同一时间只有一个 Skill 在运行。完成一个后再启动下一个。

方案三：接力棒恢复
如果接力棒已损坏，可以手动重建（Markdown 格式）：
```markdown
# 接力棒 — {任务名称}

## 元数据
- state: 分析
- project: {项目路径}
- skill_name: conspect
- mode: all
- created: {ISO 8601}
- updated: {ISO 8601}

## 状态追踪
- [ ] 开始
- [ ] 分析
- [ ] 确认
- [ ] 设计
- [ ] 设计审查
- [ ] 实现
- [ ] 验证
- [ ] 完成
```
重建后手动确认状态一致即可。

**预防措施**
- 在项目初始化时配置 `baton_file` 参数指定唯一路径
- 使用 `/conspect status` 定期检查接力棒状态
- 避免在同一终端会话中同时启动多个 Skill

## [视觉输出]

### Q9: 生成的报表能不能自定义样式？

**A:** 可以。conspect Skill 提供了多层级的自定义能力。

**样式自定义层级**

第一层：预置主题（一键切换）
在 DESIGN 阶段告诉 AI 需要的主题风格，如"使用深色主题"、"使用商务风格"等。

第二层：配置覆盖（中级自定义）
在 DESIGN 阶段可以通过配置告知具体样式参数：

```yaml
style:
  primary_color: "#2B6CB0"        # 主色
  secondary_color: "#E2E8F0"      # 辅色
  font_family: "'Microsoft YaHei', sans-serif"  # 字体
  font_size_base: 14px            # 基础字号
  chart_border_radius: 8px        # 图表圆角
  spacing: 16px                   # 间距
  responsive: true                # 自适应
  print_optimized: true           # 打印优化
```

第三层：自定义 CSS（高级自定义）
在 IMPLEMENT 阶段可以注入自定义 CSS：

```css
/* 自定义样式注入 */
.dashboard-header { background: linear-gradient(135deg, #667eea 0%, #764ba2 100%); }
.kpi-card { box-shadow: 0 10px 30px rgba(0,0,0,0.1); border-radius: 12px; }
.chart-container { padding: 20px; background: #f8fafc; }
```

第四层：模板引擎（最高级）
使用 Jinja2/Handlebars 模板完全控制输出 HTML 结构，需在 DESIGN 阶段声明使用自定义模板。

**注意事项**
- 自定义样式后建议在 VERIFY 阶段检查移动端适配
- 企业级使用建议统一维护一套品牌样式模板
- 自定义 CSS 不能覆盖数据绑定区域（图表、表格），仅控制布局和装饰

### Q10: PDF 中文乱码怎么解决？

**A:** PDF 中文乱码通常由字体缺失引起，conspect Skill 提供了完整的解决方案。

**乱码原因分析**

| 原因 | 表现 | 发生概率 |
|------|------|---------|
| PDF 生成环境缺少中文字体 | 所有中文显示为方框"[ ]" | 最高 |
| 字体未嵌入 PDF | 在查看端字体缺失时变成乱码 | 较高 |
| 编码声明错误 | 部分中文正确、部分乱码 | 中等 |
| 使用了不支持中文的字体 | 特殊字符显示异常 | 较低 |

**解决方案**

方案一：使用内置中文字体包（推荐）
在 DESIGN 或 IMPLEMENT 阶段告诉 AI"PDF 需要中文字体"，系统会自动使用内置的 Noto Sans SC（思源黑体），覆盖：
- 简体中文（GB2312 全字符集）
- 常用繁体中文
- 日文假名
- 常用标点符号

方案二：指定自定义字体
```yaml
pdf:
  font_path: "C:/Windows/Fonts/simsun.ttc"  # 使用系统宋体
  font_name: "SimSun"
  embed_fonts: true                          # 嵌入字体到 PDF
  fallback_fonts: ["Microsoft YaHei", "SimHei"]
```

方案三：使用 HTML 转 PDF（备选）
先生成 HTML，再通过浏览器内置的打印功能转为 PDF。此方式利用浏览器自身的字体渲染引擎，中文兼容性最好，但文件体积较大。

**最佳实践**
1. **开发阶段** — 告知 AI "PDF 需要中文字体"确保跨平台一致性
2. **生产交付** — 设置 `embed_fonts: true` 确保字体嵌入，避免接收方字体缺失
3. **发送给外部客户** — 使用 HTML 格式而非 PDF，彻底避免字体问题
4. **快速检查** — 在 VERIFY 阶段检查 PDF 的中文渲染是否完整

### Q11: 报表中有大量 0 值或负值怎么办？

**A:** 大量 0 值或负值会影响图表的可视化效果，conspect Skill 提供了专项处理策略。

**自动检测**
系统在 ANALYZE 阶段自动识别：
- 超过 30% 数据行为 0 值的指标列 -> 标记为"稀疏数据"
- 存在负值的指标列 -> 标记为"含负值"

**处理方式**

| 场景 | 系统处理 | 图表适配 |
|------|---------|---------|
| 大量 0 值 | 在聚合计算中排除 0 值行（求和/均值时），在标注中说明 0 值占比 | 柱状图底部显示"0 值占比 N%" |
| 少量负值（<10% 行） | 负值单独标注，分组图中负值用不同颜色区分 | 柱状图/折线图正常显示，负值段变色标注 |
| 大量负值（业务正常的盈亏对比） | 保留负值，自动切换为瀑布图或双色柱状图 | 瀑布图显示增量分解，正值/负值两色区分 |
| 全 0 值列 | 自动排除该列，在报告中标注"全 0 值，无分析价值" | 不生成图表 |

**用户干预**
- 在确认阶段可要求"排除 0 值行重新计算"
- 如需保留 0 值分析（如"零销售额月份"），在确认阶段说明即可
- 负值较多的场景建议上传时附带正负值说明文档，帮助 AI 理解业务含义

## [路径配置]

### Q12: 如何设置输出的保存路径？

**A:** conspect Skill 提供灵活的路径配置方式，支持绝对路径、相对路径和模板化路径。

**基础路径设置**

方法一：命令行参数
在实现阶段告知 AI 指定的输出目录：
```bash
# 指定输出目录
conspect render --output-dir "E:/Reports/conspect_output"

# 指定完整文件路径
conspect render --output "E:/Reports/conspect_output/经营分析报表_v1.html"

# 自动生成文件名（带时间戳）
conspect render --output-dir "E:/Reports/conspect_output" --auto-name
# 输出: E:/Reports/conspect_output/经营分析报表_20260717_103000.html
```

方法二：配置文件
```yaml
# output_config.yaml
output:
  base_dir: "E:/Reports/conspect_output"
  sub_dir_template: "{project}/{date}"
  file_name_template: "{title}_v{version}_{timestamp}"
  overwrite: false          # 文件已存在时是否覆盖
  create_sub_dirs: true     # 自动创建子目录
  backup: true              # 覆盖前是否备份旧文件
```

方法三：模板化路径（最灵活）
支持以下占位符自动替换：

| 占位符 | 示例值 | 说明 |
|--------|-------|------|
| `{project}` | sales-report | 项目名称 |
| `{date}` | 20260717 | 当前日期 |
| `{time}` | 103000 | 当前时间 |

**路径配置优先级**
1. 命令行参数（最高优先级）
2. 项目级配置文件
3. 全局默认配置
4. 工作目录（最低优先级，默认输出到当前目录）

**路径安全规则**
- 禁止输出到系统目录（如 `C:\Windows`、`C:\Program Files`）
- 禁止使用 `..` 路径穿越
- 输出目录不存在时自动创建（可在配置中关闭此行为）
- 路径中的非法字符自动替换为下划线
- 文件路径总长度限制为 260 字符（Windows）或自动使用 UNC 路径绕过
