# Conspect 执行指南

> **本文档是 Conspect 的阶段执行手册。每次激活 Skill 后，按当前状态读取对应的阶段详解执行。**
>
> **核心原则**：不做完当前阶段的强制检查点，就无法进入下一阶段。
>
> **入口检查请参考 SKILL.md**：包含完整的强制入口检查清单、状态机、防跳过/防遗忘机制。

---

## 一、激活即执行规则

当 Conspect 被激活时，AI **必须**立即执行以下流程，不得等待用户额外指令。

### 强制入口清单（硬性阻断 — 不完成不得进行任何实质性工作）

```markdown
## 🚨 强制入口清单（激活后第一步必须完成）

在回答用户任何问题、执行任何分析、写任何代码之前，必须：

- [ ] 已确定项目路径（按下方规则）
- [ ] 已执行 read {项目路径}/.agent/harness/_cs-baton.md
- [ ] 已确认接力棒存在与否
    - 存在 → 解析当前状态，准备续跑
    - 不存在 → 创建目录和接力棒，状态 = 开始
- [ ] 已执行 write {项目路径}/.agent/harness/_cs-baton.md（如果不存在）
- [ ] 已在回复**第一行**输出 "当前状态：[状态名]，下一步：[操作]"

**任何为未完成以上项目 → 禁止执行后续步骤 → 必须先完成入口清单**
**验证方式**：用户可在任何时候要求检查入口清单是否全部打勾。
```

### 项目路径确定规则

```
规则：{项目路径} 的优先级
1. 用户明确指定的路径 → 使用该路径
2. 当前工作目录（本次对话的 cwd） → 使用该路径
3. 以上都不可用时 → 使用当前工作目录作为兜底

元任务路径选择（重要）：
当用户要求审查/检查/分析 Skill 本身时（元任务）：
- 优先使用当前工作目录下的 .trae/skills/conspect/ 路径
- 如果当前工作目录下无 skills 目录，则回退到规则 1-3 的标准优先级

注意：元任务的接力棒文件依然存放在 `{项目路径}/.agent/harness/`，以保持产物路径统一。
不要让路径模糊成为跳过状态机的理由。
```

### [守卫] 首次响应守卫（最外层防线）

这是 **本 Skill 的最后一道自强制防线**，在所有规则之上：

```markdown
## [守卫] 首次响应守卫

当 Conspect 被激活时，AI 的**第一次回复**必须满足以下条件，否则视作违反本 Skill：

**条件一：回复第一行必须是** [OK]
```
当前状态：[状态名]，下一步：[操作]
```
例：`当前状态：开始，下一步：创建接力棒，进入分析阶段`

**条件二：回复中必须包含入口清单的明确执行记录** [OK]
- 不能只是在心里想"已完成"——必须写出实际执行的命令结果
- `read {项目路径}/.agent/harness/_cs-baton.md` → 显示文件内容或"文件不存在"
- `write {项目路径}/.agent/harness/_cs-baton.md`（如不存在）→ 显示写入确认

**条件三：回复中不得包含实质性工作** [OK]
（实质性工作 = 分析数据、搜索文件、写代码、修改文件、读取非入口文件的文档）
- 在确认阶段之前，**禁止执行任何不属于入口清单的操作**
- 如果用户的问题是"审查这个Skill"，第一次回复只能说"当前状态：开始，正在初始化..."
- 执行分析/搜索/读取等操作必须等开始→分析阶段

**违规检测规则**：
[WARN] 如果AI的第一次回复：
- 没有输出"当前状态"行 → 违反
- 直接开始读文件/搜索/分析 → 违反
- 输出"让我先看看文件结构" → 违反
- 使用Task子Agent执行实质性工作 → 违反
- 把入口清单在心里想一遍就当完成了 → 违反

**违反后果**：如果用户判定AI未通过"首次响应守卫"，用户有权要求AI立即停止并重新执行入口清单。

**条件性例外（修正）**：
- 状态转换（开始→分析）属于"入口清单"流程的一部分，不视为"实质性工作"
- 即首次回复可以推进 开始→分析 的状态转换和接力棒更新
- 但分析的实际工作（读数据、搜索、写分析报告）必须等第二次回复开始
```

## 四、用户中断处理机制（全阶段适用）

当用户在状态机执行过程中（非确认阶段或确认阶段）提出额外需求、问题或调整要求时，AI 必须执行以下流程：

```markdown
## 中断处理流程

1. [ ] 立即暂停当前阶段操作
2. [ ] 向用户展示 3 个选项：

   > [WARN] 我注意到您在流程执行中提出了新的需求/问题。
   > 请选择处理方式：
   > 
   > **① 立即重置**：中断当前流程，回到 分析 阶段重新分析并包含新需求
   > **② 记入 TODO**：将新需求记入接力棒"待办清单"，当前流程完成后自动重新发起任务
   > **③ 仅讨论**：继续当前任务，暂不调整或新增（仅做讨论/解答）

3. [ ] 根据用户选择执行：
   - **选项① 立即重置** → 更新接力棒状态为 分析（保留已有产物），记录中断原因和新需求，进入 分析 阶段重新分析
   - **选项② 记入 TODO** → 在接力棒新增"待办清单"章节，记录新需求，继续当前阶段/流程。当前流程完成后，自动触发"完成（新任务）"规则重新发起任务
   - **选项③ 仅讨论** → 回答用户问题/讨论后，继续当前阶段操作，不修改任务范围

4. [ ] 更新接力棒：记录中断时间、原因、用户选择
```

**注意**：此机制覆盖 AI"激活即执行"的自动推进行为。当用户主动介入时，AI 应优先响应用户中断而非自动推进。

## 五、阶段执行通用流程

每个阶段必须按以下顺序执行，**跳过任何一步都无法进入下一阶段**：

```
┌─────────────────────────────────────────────────┐
│ Step 1: 读取接力棒                               │ ← 强制
│  - 确认当前状态                                   │
│  - 确认当前模式（NORMAL/FIX/RETRY）               │
└─────────────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────┐
│ Step 2: 读取阶段指南                             │ ← 强制
│  - 读取对应的 Agent 文档                         │
│  - 读取前置产物（如 _cs-analysis.md）             │
└─────────────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────┐
│ Step 3: 执行阶段任务                             │
│  - 按 Agent 指南执行任务                         │
│  - 生成产物文件                                   │
└─────────────────────────────────────────────────┘
                   ↓
┌───────────────────────────────────────────────────────────┐
│ Step 4: 自检 + 验证链检查（自身产物验证）                  │ ← 强制
│  ├─ 检查**本次阶段产物**是否完整、格式是否符合模板          │
│  ├─ 收集"计数"证据（声明数量和实际数量对比）                │

└───────────────────────────────────────────────────────────┘

│ Step 5: 阶段转换合法性检查（前置产物验证）                   │ ← 强制阻断
│  - AI 按状态路由表校验阶段转换合法性                        │
│  - 校验接力棒状态与目标阶段是否匹配                          │

└───────────────────────────────────────────────────────────┘

│  - 修改内容：记录问题和决策                       │

│  - 验证：再次 read 确认写入成功                   │
│  【未实际写入 → 阻断 → 必须重新写入】              │
└─────────────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────────────┐
│ Step 7: 进入下一阶段                             │
│  - 按状态机流转                                   │

```

## 六、验证链规则（防虚假完成）

AI 在每个阶段的输出必须提供可验证的证据链：

- **计数验证**：声称"提取了 N 个维度/指标/图表" → 必须逐个列出 N 个项，声称 N 个但只列出 M 个(M<N) → 阻断
- **列表验证**：声称"涉及数据文件" → 必须列出具体文件路径，不得用"等"字省略
- **文件验证**：声称"已写入产物文件" → 必须执行 `read` 命令确认写入成功
- **流程图验证**：技术方案中的流程图必须使用标准 Mermaid 语法（如 `flowchart TD`）。ASCII 布局示意图在视觉设计阶段允许使用，技术方案流程图不得使用 ASCII 文字画框。
- **阻断规则**：以上任一未通过 → 视为阶段未完成 → 必须补充后继续

> **自检清单的权威来源**：每个阶段执行时，Agent 自检清单以 `artifacts/template-artifacts.md` 中对应产物的"完成后检查清单"为最终标准。

## 七、阶段详解（含强制检查点）

### 7.1 开始阶段

**入口**：用户首次触发 Conspect

**执行步骤**：

```markdown
0. [ ] 确定项目路径（元任务兜底）
   - 用户明确指定了路径？→ 使用该路径
   - 否则 → 使用当前工作目录（本次对话的 cwd）
   - 注意：即使任务是"审查 Skill 本身"也必须选一个路径
   【路径模糊不是跳过状态机的理由】

1. [ ] 读取接力棒
   read {项目路径}/.agent/harness/_cs-baton.md

2. [ ] 如果不存在，创建目录和接力棒
   mkdir -Force {项目路径}/.agent/harness/
   write {项目路径}/.agent/harness/_cs-baton.md
   （状态保持"开始"，不提前修改）

3. [ ] 识别用户需求
   - 确认数据源（Excel 文件路径、Sheet 名称）
   - 确认目标产物类型（Web看板/离线HTML/PDF/全部）
   - 确认是否有特殊要求（配色、品牌、排版偏好）

4. [ ] 自检清单
   - [ ] 接力棒已创建/已读取
   - [ ] 用户数据源已确认
   - [ ] 用户需求已记录

5. [ ] 检查点验证 — 校验阶段转换合法性
   - [ ] 接力棒文件存在
   - [ ] 接力棒包含状态字段
   - [ ] 用户数据源已明确
   【确认满足 开始→分析 条件】
   【未通过 → 阻断 → 必须修复】

6. [ ] 更新接力棒
   - 状态: 开始 → 分析
   - 记录: 用户需求摘要、数据源信息
   - 标记: 开始 [OK]

7. [ ] 第一行输出："当前状态：分析，下一步：生成数据分析报告"

8. [ ] 进入 分析 阶段
```

### 7.2 分析阶段

**入口**：开始完成 或 确认用户选择修改

**前置检查（阻断条件）**：
```markdown
- [ ] 接力棒状态为 分析 或 开始
- [ ] 项目路径已确认
- [ ] 用户数据源已明确

**如果任一不满足 → 阻断 → 返回 开始 阶段**
```

**执行步骤**：

```markdown
1. [ ] 读取 Analyzer Agent 指南
   read {Skill路径}/agents/analyzer-agent.md

2. [ ] 读取并解析数据源
   - 遍历用户提供的 Excel 文件
   - 对每个文件：使用 `pd.ExcelFile` 枚举所有 Sheet 名称 → 记录完整 Sheet 列表
   - 确认分析范围覆盖所有 Sheet，如有排除需注明原因
   - 逐 Sheet 解析表头和数据行
   - 数据类型检测（数值/日期/文本/布尔）
   - 数据质量检查：空值统计、重复行检测、类型异常检测

3. [ ] 执行维度分析和指标计算
   - 确定业务维度（时间、分类、地域等）
   - 确定度量指标（销售额、数量、增长率等）
   - 执行聚合计算（SUM/AVG/COUNT/MAX/MIN）
   - 计算衍生指标（同比、环比、占比、累计）
   - 输出维度分析表和指标汇总表

4. [ ] 生成分析报告
   - 写入 {项目路径}/.agent/harness/_cs-analysis.md
   - 按照 artifacts/template-artifacts.md 的"分析报告模板"

5. [ ] 自检清单
   以 `artifacts/template-artifacts.md` 中 _cs-analysis.md 的"完成后检查清单"为最终标准
   - [ ] 产物完整、格式正确、内容完备
   - [ ] **Sheet 枚举完整：分析报告中已列出所有 Sheet，无遗漏**
   - [ ] 数据维度分析覆盖所有字段
   - [ ] 指标计算结果正确（可溯源到原始数据）
   - [ ] 验证链通过（计数/列表/文件验证）

6. [ ] 质量审核检查点（阻断点）
   - [ ] 拉起质量审核 Agent 审核 _cs-analysis.md
   - [ ] 审核通过 → 继续
   - [ ] 审核不通过 → 读取审核报告的"待修复问题清单" → 返回修复
   - [ ] 写入审核报告文件：_cs-qa-analysis.md
   - [ ] 更新接力棒 quality_audit_analysis 状态

   **未通过 → 输出错误："分析产物质量审核不通过，详见 _cs-qa-analysis.md" → 必须修复**
   **重试超过2次 → FAILED → 需人工介入**

7. [ ] 检查点验证（阻断点）
   - [ ] 产物文件存在
   - [ ] 格式符合模板
   - [ ] 内容完整（所有必需章节）
   - [ ] _cs-qa-analysis.md 存在（质量审核已通过）

   **未通过 → 输出错误："分析产物不完整，缺少：[具体项]" → 必须修复**

8. [ ] 更新接力棒
   read {项目路径}/.agent/harness/_cs-baton.md
   - 状态: 分析 [OK] → 确认
   - 下一步: 确认
   - quality_audit_analysis 行: 将"状态"列改为"已通过"，"报告文件"列填入 _cs-qa-analysis.md
   write {项目路径}/.agent/harness/_cs-baton.md

9. [ ] 进入 确认 阶段
```

---

### 7.3 确认阶段

**入口**：分析完成

**[WARN] 关键**：这是唯一的用户交互节点！**必须等待用户明确响应，不能自动继续！**

**前置检查（阻断条件）**：
```markdown
- [ ] _cs-analysis.md 存在（执行：ls {项目路径}/.agent/harness/_cs-analysis.md）
- [ ] _cs-analysis.md 内容完整（执行：read {项目路径}/.agent/harness/_cs-analysis.md | head -20）
- [ ] _cs-qa-analysis.md 存在（质量审核已通过）

**如果不存在 → 阻断 → 返回 分析 阶段**
```

**执行步骤**：

```markdown
1. [ ] 读取 _cs-analysis.md
   命令：read {项目路径}/.agent/harness/_cs-analysis.md

2. [ ] 展示摘要（必须包含以下内容）
   
   ---
   
   ## [清单] 数据分析摘要
   
   > 请花 1 分钟确认数据分析结果是否正确。
   
   ### 数据源概览
   
   | # | 文件 | Sheet | 行数 | 列数 | 关键字段 |
   |---|------|-------|------|------|---------|
   | 1 | {文件1} | {Sheet} | {N} | {N} | {关键字段} |
   
   ### 维度与指标
   
   | 维度 | 字段 | 取值数 | 说明 |
   |------|------|--------|------|
   | 时间 | {字段} | {N} | {说明} |
   | 分类 | {字段} | {N} | {说明} |
   
   | 指标 | 聚合方式 | 数值 | 同比/环比 |
   |------|---------|------|----------|
   | {指标} | {sum/avg} | {值} | {%} |
   
   ### 图表推荐
   
   | # | 图表类型 | 数据来源 | 说明 |
   |---|---------|---------|------|
```

## 八、修复回路

当验证阶段发现问题时，按以下策略路由修复：

| 错误类型 | 策略 | 计入重试 | 说明 |
|----------|------|-----------|------|
| DESIGN_ISSUE | 返回 设计（修复） | [OK] design_fix_retry | 图表选型、排版布局等设计问题，最多修复2次 |
| DATA_MISMATCH | 返回 实现（重试） | [OK] retry | 数据绑定错误、图表渲染错误，最多2次 |
| RENDER_FAILURE | 返回 实现（修复） | [禁止] | 渲染代码错误或文件生成失败 |
| ENVIRONMENT | 报告用户，等待处理 | [禁止] | 需人工介入（如依赖缺失、权限问题） |

### 重试超限处理

当修复循环达到重试上限时：
```markdown
1. 更新接力棒状态为 FAILED
2. 输出失败报告
3. 记录失败原因到接力棒
4. 建议人工介入
5. AI 提供修复建议和已完成的中间产物
```

## 附录A：终止报告模板

### A.1 完成报告（流程成功完成）

```markdown
# [OK] Conspect 流程完成报告

## 项目信息
- 项目路径: {项目路径}
- 流程时间: {开始时间} ~ {结束时间}

## 任务完成清单

| # | 阶段 | 状态 | 产物 | 说明 |
|---|------|------|------|------|
| 1 | 开始 | [OK] | _cs-baton.md | 启动 |
| 2 | 分析 | [OK] | _cs-analysis.md / _cs-qa-analysis.md | 分析通过 |
| 3 | 确认 | [OK] | - | 用户确认 |
| 4 | 设计 | [OK] | _cs-design.md / _cs-qa-design.md | 设计通过 |
| 5 | 实现 | [OK] | _cs-implement.md / _cs-qa-implement.md + 渲染产物 | 实现完成 |
| 6 | 验证 | [OK] | _cs-verify.md / _cs-qa-verify.md | 验证通过 |
| 7 | 完成 | [OK] | _cs-report.md | 流程完成 |

## 最终产物
- 全部产物位于 `{项目路径}/.agent/harness/`

## 总结
- 总耗时: {时长}
- 重试次数: {n}
- 质量评分: {评分}
```

### A.2 终止报告（用户取消）

```markdown
# [终止] Conspect 流程已终止

## 项目信息
- 项目路径: {项目路径}
- 流程时间: {开始时间} ~ {终止时间}
- 终止阶段: {终止时的状态名}
- 终止原因: 用户选择取消

## 已完成的工作

| # | 阶段 | 状态 | 产物 |
|---|------|------|------|
| 1 | 开始 | [OK] | _cs-baton.md |
| 2 | 分析 | [OK] | _cs-analysis.md |
| ... | ... | ... | ... |

## 说明
流程因用户取消而终止。产物保留在 `{项目路径}/.agent/harness/` 中可供参考。
如需重新启动，可手动设置接力棒状态并重新开始。
```

```markdown
## 项目信息

- 流程时间: {开始时间} ~ {失败时间}
- 失败阶段: {失败时的状态名}

## 失败原因

```
{描述失败的详细原因：如重试超限、阻断检查未通过等}
```

## 已完成的产物

| # | 阶段 | 状态 | 产物 |
|---|------|------|------|
| 1 | 开始 | [OK] | _cs-baton.md |

## 恢复建议

3. 如问题无法解决，请联系项目管理员

## 错误日志
{相关错误信息或日志输出}
```
